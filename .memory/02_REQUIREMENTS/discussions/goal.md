사용자 편의성 추가

1. 프로젝트를 만든 이유
- 모든 필요 문서가 존재하면 해당 문서 기반으로 LLM이 프로젝트를 자동으로 작성하는 것은 가능하다
- 문서는 사람이 읽고 이해하기 쉬운 형태로 작성되어야 한다(여러 사람이 봐도 같은 의미로 이해할 수 있어야 한다, 필요하면 수정도 가능한 형태여야 한다)

2. 프로젝트 프로세스
- CQ를 생각하여 프로젝트에서 필요한 기능들을 정리한다
- 정리되지 않은 문서들을 요구사항 문서로 구조적으로 정리한다 
- 요구사항 문서 기반으로 구현해야 할 plan(실행 문서)을 작성한다
- plan 기반으로 프로젝트를 구현한다
- 구현이 완료되면 실행 문서를 아카이빙한다

3. 사용자 편의성 고려
- 프로세스의 이름을 명확하게 지정하고 설명하는 문서가 필요함
- 해당 프로세스에서 사용자가 검토해야 할 부분에서만 검토하도록 하고 llm이 자동으로 진행해야 할 부분은 자동으로 수행되도록 해야 한다(mcp사용)
- cq기반 기능 문서는 유저가 작성(혹은 llm과의 토의로 작성), 문서의 규칙은 존재하지 않음, 꼭 문서화 하지 않고 바로 진행해도 되어야 함

3.1 일반 프로세스
- 해당 사용자가 생각하는 필요 기능을 생각해냈을 때 해당 요구사항 문서 업데이트하라고 하면 요구사항을 분리해서 규칙대로 여러 요구사항 문서로 업데이트한다
- 그 다음 요구사항 기반으로 실행 문서 생성하라고 하면 실행 문서를 작성한다
- 실행 문서 기반으로 프로젝트 구현하라고 하면 프로젝트를 구현한다
- 구현이 완료되면 실행 문서를 아카이빙한다

3.2 MCP 기반 프로세스
- 해당 사용자가 생각하는 필요 기능을 생각해냈을 때 해당 요구사항 Intake 라고 llm cli에게 이야기했을 때 해당 요구사항을 분리해서 규칙대로 여러 요구사항 문서로 업데이트하고 해당 요구사항들을 모은 brief 문서를 작성한다
- 다음 사용자가 해당 brief 문서를 검토하고 plan을 cli에게 명령했을 때 실행 문서를 작성한다
- 다음 사용자가 해당 실행 문서를 검토하고 run을 cli에게 명령했을 때 실행 문서 기반으로 프로젝트 업데이트 후 실행 문서 완료 확인 후 아카이빙한다
